WalrusVault — Decentralized Digital Legacy Platform
====================================================
Tatum x Walrus Hackathon Submission

CONCEPT
-------
WalrusVault is a "Dead Man's Switch" for your digital life. You upload important
documents (wills, seed phrases, private keys, passwords, legal docs) encrypted
with AES-256 into Walrus decentralized storage. If you stop checking in for a
set period (30-3650 days), your designated recipients automatically gain access
to decrypt and download everything.

WHY THIS MATTERS
----------------
Every year, billions of dollars in crypto are lost forever because holders die
or become incapacitated without passing on their keys. Traditional solutions
require trusting lawyers, centralized services, or family members with sensitive
data. WalrusVault eliminates ALL trust assumptions:

  - Documents are encrypted CLIENT-SIDE before upload — server never sees plaintext
  - Storage is on Walrus (decentralized, permanent, censorship-resistant)
  - Vault state lives on Sui blockchain (trustless, transparent)
  - Dead Man's Switch logic is enforced by smart contract, not a company
  - Recipients can only access AFTER timeout expires — zero premature access

HOW IT WORKS — USER FLOW
-------------------------

Step 1: CREATE VAULT
  - Connect Sui wallet (Sui Wallet, Suiet, etc.)
  - Upload documents (drag & drop, any file type)
  - Each file is encrypted with AES-256-GCM using a PBKDF2-derived key
    from a user-provided passphrase (100K iterations, random salt)
  - Encrypted blobs are uploaded to Walrus via REST API
  - Walrus returns a blob_id — this is stored on-chain

Step 2: CONFIGURE VAULT
  - Set timeout period (e.g., 90 days)
  - Add recipients (up to 10 Sui wallet addresses)
  - Optional: set recipient shares (equal split by default)
  - All metadata written to Sui Move smart contract

Step 3: CHECK-IN (Ongoing)
  - Vault owner must "check in" before the deadline
  - Check-in resets the countdown timer
  - Tatum API monitors wallet activity — any outgoing tx = implicit check-in
  - Manual check-in also available via dashboard button

Step 4: RELEASE (If owner stops checking in)
  - When timeout expires, vault enters "claimable" state
  - Recipients see the vault in their dashboard
  - They claim their share → receive the Walrus blob_ids + decryption passphrase
  - Passphrase is split via Shamir's Secret Sharing (2-of-3 threshold)
  - At least 2 recipients must cooperate to fully decrypt

  - Blob upload: PUT /v1/blobs/{base64_data}
  - Blob download: GET /v1/blobs/{blob_id}
  - Data is erasure-coded and distributed across storage nodes

Encryption:
  - AES-256-GCM (client-side, browser Web Crypto API)
  - PBKDF2 key derivation (100,000 iterations, SHA-256)
  - Random 16-byte salt + 12-byte IV per file
  - Passphrase never leaves the browser

Monitoring:
  - Tatum API for wallet activity monitoring
  - Auto check-in: detects outgoing transactions as "proof of life"
  - Polling interval: every 5 minutes
  - Fallback: manual check-in via dashboard

Smart Contract (vault.move)
---------------------------

Structs:
  - Vault: owner, recipients[], timeout_days, last_checkin, blob_ids[],
           passphrase_hash, status (active/claimed/expired)
  - VaultCreated event
  - VaultClaimed event
  - VaultCheckIn event

Functions:
  - create_vault(recipients, timeout_days, blob_ids, passphrase_hash)
    → Creates a new Vault object owned by sender

  - check_in(vault_id)
    → Resets last_checkin timestamp. Only callable by owner.

  - claim_vault(vault_id)
    → Recipient claims access. Only callable after timeout expires.
    → Emits VaultClaimed with blob_ids + passphrase reference.

  - add_recipient(vault_id, recipient)
    → Owner adds a recipient. Max 10.

  - remove_recipient(vault_id, recipient)
    → Owner removes a recipient before expiry.

  - extend_timeout(vault_id, additional_days)
    → Owner extends the timeout period.

Security Model
--------------
1. Encryption happens in-browser — encrypted bytes go to Walrus, never plaintext
2. Passphrase is derived from user input + random salt, stored nowhere
3. Smart contract enforces timeout — no admin override, no backdoor
4. Recipients need cooperation (Shamir 2-of-3) to decrypt — single compromised
   recipient cannot steal data
5. Tatum auto-check-in is a convenience layer — doesn't weaken the security model
   because the blockchain is the source of truth for timeout state
6. Walrus storage is permanent and censorship-resistant — data survives even if
   the frontend goes down (blob_ids are on-chain)

Hackathon Differentiators
--------------------------
1. True Decentralization — no centralized backend, no database, no server state
2. Client-side encryption — zero-knowledge architecture
3. Dead Man's Switch with auto check-in via Tatum — unique UX that prevents
   false triggers while keeping the owner safe
4. Multi-recipient with Shamir's Secret Sharing — cooperative decryption
5. Walrus for permanent storage — data survives forever, not just until
   a company shuts down
6. Production-ready UI — not a hackathon prototype, but a polished product

Known Limitations (Hackathon Scope)
------------------------------------
- Shamir's Secret Sharing is simplified for demo (full implementation pending)
- Tatum auto-check-in polling is simulated (production would use webhooks)
- No mobile-optimized UI yet
- Smart contract doesn't have formal audit
- Single-chain (Sui only) — cross-chain expansion planned
